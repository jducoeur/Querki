# Querki MySQL Usage Inventory
## Purpose: MySQL 5.7 → 8.4 upgrade preparation

Generated by reading all Anorm-using source files. All paths are relative to `querki/`.

## Database Infrastructure

### `scalajvm/app/querki/db/ShardKind.scala`
Defines the four logical DB shards and their Play config names:
- `System` → `"system"` — global user/identity/space registry
- `User` → `"user"` — per-node user data: space thing tables, conversations, user values, notifications
- `Template` → `"template"` — read-only template DB used only during functional test setup
- `Test` → `"test"` — functional test databases

### `scalajvm/app/querki/db/QDB.scala`
Thin wrapper around Play's `DBApi.database(name).withTransaction`. Every DB call in the codebase goes through `QDB(ShardKind.X) { implicit conn => ... }`. Logs exceptions and re-throws.

### `scalajvm/app/querki/util/SqlHelpers.scala`
Anorm helper utilities: `oid(name)` row parser returning `OID`, plus deprecated `EnhancedSqlRow` implicit for old-style row access.

### `scalajvm/app/querki/time/TimeAnorm.scala`
Custom Anorm type conversions for Joda DateTime ↔ `java.sql.Timestamp`. `dateTimeToStatement` zeroes milliseconds on write (`.withMillisOfSecond(0)`). Used implicitly throughout all persistence files.

## Key Observations for MySQL 8.4 Upgrade

### Definite action items

1. **`com.mysql.jdbc.exceptions.jdbc4.MySQLSyntaxErrorException`** (NotificationPersistence.scala line 107): This catches the old Connector/J exception class path. MySQL 8 ships with Connector/J 8.x where the exception hierarchy has moved. The new path is `com.mysql.cj.jdbc.exceptions.MysqlDataTruncation` / the base is still `java.sql.SQLSyntaxErrorException`. Should catch the JDBC standard `java.sql.SQLSyntaxErrorException` instead.

2. **`DEFAULT CHARSET=utf8`** in all four CREATE TABLE DDLs (Step4, Step5, UserStep1, and SpaceManagerPersister's thing-table creation): MySQL 8.0+ deprecated the `utf8` alias and in 8.4 it still maps to `utf8mb3`. For correctness these should be `utf8mb4`. However, these are evolution steps that run on old Spaces — changing them would not retroactively fix existing tables. New-table creation in SpaceManagerPersister (line 103) affects new Spaces going forward.

3. **`information_schema` query** (SpacePersister.scala line 183): The query `SELECT COUNT(*) FROM information_schema.tables WHERE table_name = '{tname}'` doesn't filter by `table_schema`. In MySQL 5.7 this worked because there was typically only one user DB. In production the behavior should be the same, but it's worth adding `AND table_schema = DATABASE()` for correctness.

4. **`INSERT Identity (...)` without `INTO`** (UserPersistence.scala line 411): `INSERT Identity ...` without `INTO` is a MySQL extension that has historically worked but is non-standard. MySQL 8 still supports it, but worth noting.

5. **`bigInt` casing** (Step4.scala line 18): `authorizedBy bigInt DEFAULT NULL` — `bigInt` with capital I. MySQL has historically been case-insensitive for type names, and 8.4 should be fine, but worth normalizing to `bigint`.

### Likely non-issues but worth verifying

6. **`BOOLEAN`/`TINYINT` storage**: MySQL stores BOOLEAN as TINYINT(1); Anorm reads it back as Int. The `bool()` parser in SqlHelpers handles this. MySQL 8 behavior is unchanged.

7. **`MEDIUMTEXT`**: Maximum 16 MB. `max_allowed_packet` must accommodate this. Default in MySQL 8 is 64MB (up from 4MB in 5.7), so this should be less of a concern but verify the production setting.

8. **Anorm version**: The codebase uses Anorm 2.x (Play 2.8). Anorm is not MySQL-version-sensitive; it uses JDBC which is stable. No Anorm changes expected.

9. **No `GROUP BY` with non-aggregated columns**: The queries avoid MySQL 5.7's `ONLY_FULL_GROUP_BY` issues that MySQL 8 enforces by default. The one `GROUP BY`-like query (`getAcquaintanceIds`) uses `DISTINCT` instead, which is fine.

10. **Cross-DB transactions**: Multiple comments in the code note that operations spanning System + User DBs cannot be in a single transaction. This is unchanged in MySQL 8.

11. **Dynamic table names**: `{tname}`, `{cname}`, `{uvname}`, `{notename}` are interpolated as raw strings (not Anorm parameters) because parameterized table names don't work in MySQL. Table names are derived from OIDs (controlled internal values), not user input. No SQL injection risk, no MySQL 8 change needed.
